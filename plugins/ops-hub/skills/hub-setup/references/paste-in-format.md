# Ops Hub setup — paste-in format

The contract between the setup web app and `hub-setup`. The app emits this; the skill parses it. Keep both sides in sync with this file.

## Envelope

The user pastes a short prompt containing one fenced `json` block:

````
Run the hub-setup skill. Apply this Ops Hub setup. (Generated by the Ops Hub setup app; review before approving.)

```json
{ ...setup object... }
```
````

`hub-setup` triggers on the "Run the hub-setup skill" / "Apply this Ops Hub setup" lines plus the `opsHubSetup` marker inside the JSON.

## The setup object

```json
{
  "opsHubSetup": true,
  "baseVersion": 1,
  "records": {
    "client":   { "fields": { }, "body": [ ] },
    "project":  { "fields": { }, "body": [ ] },
    "task":     { "fields": { }, "body": [ ] },
    "pipeline": { "fields": { }, "body": [ ] }
  },
  "renames": {
    "client":   [ { "from": "📋 Status", "to": "Stage" } ],
    "project":  [],
    "task":     [],
    "pipeline": []
  },
  "recordRenames": {
    "client": { "from": "Clients", "to": "Customers" }
  }
}
```

- **`records`** — the full *target* shape per record type: `fields` (meaning-key → field entry) and `body` (ordered sections, each `{ heading, purpose }`). The end-state the team wants, defaults included for anything untouched. **Source locations are not part of the app's output** — see *Source locations* below.
- **`renames`** — per record type, the label changes the user made, so the skill applies a Notion **RENAME** (preserves data) instead of drop+add. `from` = current live label, `to` = desired label.
- **`recordRenames`** — database (data-source) name changes, keyed by the fixed record key whose title changed: `{ from, to }`. Only changed records appear (omit unchanged). The skill renames the Notion **database title** (not a field). Safe by design: skills resolve the four tables by role, never by name, and the config stores no database names — so a rename touches nothing but the title. `from` = current live database title, `to` = desired.
- **`baseVersion`** — the `config.default.json` version the app generated from. Lets the skill warn if the live config has already moved past it (the app only knows the shipped defaults).

## Field entry

```json
"status": { "label": "Stage", "purpose": "Engagement state…", "type": "select" }
```

- **`label`** — the Notion field name the team wants (may include emoji).
- **`purpose`** — the meaning, written into the config page (never into Notion).
- **`type`** — required only for **added** fields, so the skill creates the right Notion type. One of: `select`, `multi_select`, `rich_text`, `number`, `date`, `url`, `email`, `phone`, `checkbox`, `people`, `files`. Existing fields keep their live type unless explicitly retyped.

## Body section entry

```json
{ "heading": "👥 Contacts", "purpose": "Who the client's people are…" }
```

- **`heading`** — the section heading the skill composes under (may include emoji).
- **`purpose`** — what goes in the section: the skill composes to it, and the team reads it to understand and customise. The last body entry is always Manual Notes.

## How the skill maps it to actions

| Action | Detected when |
|---|---|
| **RENAME** | the pair appears in `renames` |
| **Database RENAME** | the record key appears in `recordRenames` (renames the data-source title; never a drop/recreate) |
| **ADD** | a `fields` entry whose `label` matches no live field (after applying renames) |
| **DROP** | a live *ordinary, writable* field whose label is in no `fields` entry and no `renames.from` (⚠ deletes data — confirmed separately; read-only/formula fields are never dropped) |
| **Config-only** | a changed field `purpose`, a body section's heading or meaning, or `body` order |

**Structural fields** (titles, relations) never appear as drops or retypes — the app omits them from the editable set. The same goes for **read-only / formula fields** (e.g. a Weighted Value formula). If one shows up anyway, the skill refuses that single item and continues.

**Form views.** A DROP also removes the field from any Notion **form view** that uses it — the bound question disappears silently. `hub-setup` reads each affected table's forms live and names the impact in the drop preview (see SKILL.md). The app's output carries no form data; this is resolved entirely at apply time.

## Source locations

The app does **not** edit `sources` — that concept is kept out of the setup UI. On first setup, `hub-setup` seeds each record's `sources` from `config.default.json` when it writes the config page; thereafter they're left untouched, edited directly in the `⚙️ Hub Config` page if a team needs to. The enrichment skills (`client-write`, `project-write`) read them from there.

## Notes for the web app

- Start from `config.default.json`, so the emitted `records` is always complete (defaults for anything the user didn't touch).
- Track `renames` as the user edits a label in place (don't emit a rename if `from == to`).
- Track `recordRenames` likewise as the user edits a database name (don't emit if `from == to`); the record key stays fixed (`client` / `project` / `task` / `pipeline`).
- No Notion access needed — the default shape is known up front; the skill does all live work.
